📌 概述

本手册面向使用腾讯 Workbuddy（类似 Claude Cowork 的 AI 工作客户端）的用户。在对话框里导入文件、选技能、发需求，等结果即可——不用命令行。meetings 技能把会议转录文本自动转成正式会议纪要 Word 文档。

💡
Workbuddy 操作简单、开箱即用；但可选模型少、不支持自定义模型、速度较慢。若你需要自定义模型、批量处理或更快速度，请看 Claude Code 版手册。两者装好技能后用法一致。

📲 下载客户端并登录

1

下载 Workbuddy

访问 copilot.tencent.com/work 下载客户端并安装。

2

微信扫码登录

打开客户端，用微信扫码登录即可，无需额外账号或 API Key。

🧩 下载并导入技能

meetings（出纪要）和 iTingnao（音频转写）都是技能。先从网站下载，再导入 Workbuddy。

1

下载技能

访问 GitHub 仓库 github.com/ricym/AI-meeetings-cpcec，点击 Code → Download ZIP 下载，解压后得到 meetings 技能目录（含 SKILL.md、config.json 等）。

2

导入 Workbuddy

打开 Workbuddy，在对话框点击 技能 → 导入技能，把解压出的 SKILL.md 内容粘贴进去即可。

3

（音频才需要）导入 iTingnao 技能

要处理音频文件时，同样方式导入 iTingnao 技能并配置 API Key。只有文本文件则跳过。

⚙️ 配置

Workbuddy 同样读取本地文件，配置方式与 Claude Code 一致。你唯一要在首次确认的是 config.json 里的 project_root。

✅
业务人员：只需把下方 project_root 改成你自己的 ai-meeting 路径，其余保持不动。知识库三件套放在该目录的 profiles/ 下。
1

编辑 config.json

确保技能的 config.json 中 project_root 指向你的 ai-meeting 目录：

{ "project_root": "C:\\Users\\你的用户名\\Documents\\ai-meeting" }
⚠️
其余配置（registry.json / profile.json / profiles/ 目录）关系到模板与脚本匹配，不要轻易改动。

📝 使用 meetings

对话框操作：导入文件 → 选技能 → 发需求 → 等结果。Workbuddy 没有命令行参数，靠自然语言描述会议类型。

📌 起点 1：文本文件（不用 iTingnao）

对话框导入已转写的文本稿 → 选 meetings 技能 → 发送需求：

这是例会，按例会出纪要

靠自然语言识别会议类型，效果等同 Claude Code 的自然语言用法。

📌 起点 2：音频文件（一条指令调两次技能）

对话框导入音频 → 选 iTingnao + meetings 两个技能 → 一条需求里直接说先转写再出纪要：

先转写这段录音，再按例会出会议纪要

iTingnao 完成转写后，大模型会接着用 meetings 生成纪要，无需手动分两步。

✅ 核对归档

1

查看输出结果

等待大模型返回会议纪要及来源标注（每条信息的来源时间戳与原文，可追溯）。

2

核对「待确认」字段

纪要中标注「[待确认]」的字段（如日期推断、疑似责任人）需人工核实后定稿。

❓ 常见问题

处理速度较慢？

Workbuddy 处理速度相对 Claude Code 较慢，复杂会议请耐心等待；急用或批量可改用 Claude Code。

技能没生效？

确认已通过「技能 → 导入技能」导入 SKILL.md，并在发送需求前选中对应技能。

音频转写失败？

音频需同时选中 iTingnao 技能并已配置 API Key；纯文本纪要则不需要。
