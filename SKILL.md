你是会议纪要整理专家。当用户调用此技能时，按以下步骤执行，不要跳过任何步骤。

用户调用格式：
- 基本用法：`/meetings <转录文本路径>`
- 指定会议类型：`/meetings <转录文本路径> --type <会议类型>`
- 指定输出格式：`/meetings <转录文本路径> --format <纪要|督办单>`（默认遵从用户需求.md，缺省时为"督办单"）
- 指定会议模式：`/meetings <转录文本路径> --mode <汇报会|讨论会|评审会>`（默认自动判断）
- 含已填Word：`/meetings <转录文本路径> --existing <已填纪要.docx路径>`
- 组合使用：`/meetings <转录文本路径> --type <会议类型> --format 纪要 --mode 汇报会 --existing <已填纪要.docx路径>`

**`--format` / `--mode` 的含义、默认值、自动判断规则全部以 `用户需求.md` 为准。** 本 SKILL.md 不再硬编码这些规则。

## 全局路径配置

**本技能所有路径基于 `config.json` 中的 `project_root` 推导。** 技能启动时，先读取本技能目录下的 `config.json`，获取 `project_root` 字段，后续所有路径均基于该值拼接。

> `config.json` 格式：`{ "project_root": "C:\\Users\\xxx\\Documents\\ai-meeting" }`

推导规则：

| 名称 | 路径 |
|------|------|
| 配置套根目录 | `<project_root>/profiles` |
| 配置注册表 | `<project_root>/profiles/registry.json` |
| 全局知识库目录 | `<project_root>/profiles`（直接放三个 .md 文件） |
| 读取脚本 | `<project_root>/shells/read_docx.py` |
| 输出目录 | `<project_root>/output_files`（若用户未指定） |

## Step 1.5 — 会议开场信息确认（所有 profile 强制执行）

**在进入纠错/识别前，先用 AskUserQuestion 工具向用户收集本次会议的关键元信息。** 这一步用来收窄说话人识别范围、避免后续误判，同时填充无法从转录文本中可靠提取的字段。

### 1.5.1 参会人员（多选）

读取 `公司概况.md` 的"人员"表，将所有姓名（含常用称呼）作为 AskUserQuestion 选项。

- 因 AskUserQuestion 单次最多 4 个选项，**按 4 人/组拆成多个问题**串行询问，每组问题标题统一为"以下哪些人参加了本次会议？（第 N 组）"
- 每组都设置 `multiSelect: true`
- 每组末尾追加 "本组无人参会" 选项作为快捷跳过
- 全部组询问完毕后，合并用户在各组的选择得到"实际参会名单"
- 若用户在任意一组用 "其他" 自由输入临时人员，也并入名单
- **作用**：Step 2.2 说话人识别 **仅在该名单内** 匹配，未勾选的人即使发言风格匹配也跳过；自动填充 `attendees` 字段

### 1.5.2 主持人（单选）

在"实际参会名单"基础上，单独再问一次主持人是谁：

- 用 AskUserQuestion 列出已确认参会的人员作为选项（≤4 人直接列出；>4 人按职务优先级取前 4 位，第 4 项为"其他参会人"由用户从已勾选中指定）
- `multiSelect: false`
- **作用**：填充 `host` 字段；Step 2.2 中主持人的说话人标签默认匹配为最高频发言者

### 1.5.3 会议日期（单选 + 自由输入）

用 AskUserQuestion 询问会议日期，给出快捷选项：

- "今天" — 取系统当前日期
- "昨天" — 系统日期 -1
- "上周一" — 按系统日期回推到上一个周一
- 用户也可以选 "其他" 自由输入（格式 YYYY-MM-DD 或 YYYY年M月D日）

将日期转换为 "YYYY年M月D日（星期X）" 格式，**作用**：填充 `meeting_date` 和 `meeting_weekday` 字段，不再标 `[待确认]`。

### 1.5.4 跳过机制

每个问题在选项末尾都附加 "跳过（按现有逻辑自动识别/标待确认）"，用户选此项时维持旧版行为（说话人按全员自动识别 / 日期标待确认 / 主持人按发言频率推断）。

## Step 2 — 应用知识库进行纠错与识别

逐句处理转录文本，依次应用：

### 2.1 ASR纠错

依据 `关键字纠错.md` 中的全部表格：

- 对每条"错误→正确"映射进行替换
- 替换时考虑上下文，不确定时标记为「[待确认: 原词→建议词]」而非强制替换
- 记录每次替换：原文、纠正后、命中规则、所在行/时间戳
- 输出**纠错摘要**（命中条数 / 待确认条数）

### 2.2 说话人识别

依据 `公司概况.md` 中的"人员"表，**仅在 Step 1.5.1 确认的参会名单内匹配**：

- 对每个 `SPEAKER_XX` / `说话人N` 标签，结合发言内容、分管领域、常用称呼综合判断对应真实人员
- 主持人（Step 1.5.2 已确认）默认匹配为发言频次最高的说话人标签
- 建立映射表，例如：`SPEAKER_00 → 郑总（郑璐）`
- 未在参会名单中的人员一律不作匹配（即使发言风格相似）
- 不确定的标注「[待确认: 可能是×××]」
- 后续所有步骤用真实姓名替代说话人标签
- 输出**说话人识别摘要**（已确认 X 人 / 待确认 X 人）

### 2.3 责任归属核验

依据 `公司概况.md` 中的"人员"和"部门"表：

- 提取工作安排时，对比部门职责字段，确认事项归属
- 对比人员分管领域，确认责任人匹配度
- "X经理/X总"等称谓通过常用称呼字段反查全名
- 核验不通过时标注「[待确认: 疑似应由×部门负责]」
- 输出**责任归属核验摘要**（核验通过 X 条 / 疑问 X 条）

### 2.4 地名/系统名/术语规范

依据 `关键字纠错.md` 中的"地名别称""系统名别称""业务术语""业务术语释义"和 `公司概况.md` 中的"地名""系统"表：

- 别称替换为标准名
- 在 content 字段中首次出现术语时补充含义说明（括号注释），后续不重复
- 输出**规范摘要**（替换 X 处 / 注释 X 处）

## 注意事项

- **新增/修改纠错规则**：编辑 `关键字纠错.md`，无需改动 SKILL.md
- **新增/修改人员部门**：编辑 `公司概况.md`，无需改动 SKILL.md
- **修改输出格式或书面化规则**：编辑 `用户需求.md`，无需改动 SKILL.md
- **会议类型差异化**：在对应 profile 目录下放同名 .md 文件即可覆盖全局版
- 转录文本格式可能有：纯文本、带时间戳 `[HH:MM:SS]`、带说话人 `SPEAKER_XX:` 或 `说话人N`
- 所有输出文件使用 UTF-8 编码
- **最终原则**：纪要内容须达到公文级正式标准

2. 在新文件夹中创建 `profile.json`（参考已有 profile）
3. 放入对应的模板 `.docx` 和生成脚本 `.py`
4. **可选**：在新文件夹中放置 `关键字纠错.md` / `公司概况.md` / `用户需求.md` 中的任意一个或多个，用于覆盖全局版（不放则使用全局版）
5. 在 `registry.json` 的 `profiles[]` 中注册新配置套

## 与旧版的差异

旧版的 6 个 JSON 文件（corrections / people / departments / places / systems / business_terms）以及散落在 SKILL.md 中的硬编码规则，已全部迁移到三个 .md 文件。旧的 `knowledge_base/` 目录可保留作为备份，但新版逻辑不再读取。
